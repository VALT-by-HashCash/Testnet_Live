# Valt Testnet 2.0 ŌĆö Code Analysis & Improvement Recommendations

Generated by reviewing all component, store, service, and router source files.
See `src/test/edge-cases/edge-cases.test.js` for tests that exercise many of these findings.

---

## Critical Bugs

### 1. `VerifyTestnet.vue` ŌĆö Null-check does not catch `{ user: null }` API response
**File:** `src/components/VerifyTestnet.vue`
**Severity:** High
**Issue:** The user extraction chain `res?.data?.user ?? res?.data ?? res` falls through to `res?.data`
when the API returns `{ data: { user: null } }`. The `data` object `{ user: null }` is truthy,
so `if (!user)` never throws and the component continues as if the user is valid, silently failing.
**Fix:** Change the null-check to `if (!user?.email)` or explicitly check `res?.data?.user !== null`.

### 2. `api.js` ŌĆö `http.create()` called on an axios instance
**File:** `src/services/api.js`
**Severity:** High
**Issue:** `http` is an axios *instance* (returned from `axios.create()`). Axios instances do not
have a `.create()` method ŌĆö only the top-level `axios` object does. This would throw a
`TypeError: http.create is not a function` at import time in any environment that actually
executes this code path.
**Fix:** Use `axios.create(...)` directly, or restructure so the base instance is exported and
used directly.

### 3. `valt.js` store ŌĆö Dashboard service methods are stubs
**File:** `src/stores/valt.js` + `src/services/dashboard.js`
**Severity:** High
**Issue:** `loadDashboardData()` calls `dashboardService.getNotifications()`,
`getProfileDetails()`, `calculateUserStats()`, `getRewards()`, and `getTrafficStats()` ŌĆö none
of which exist in the real `dashboard.js` file (it is a minimal stub). Every call will throw
`TypeError: dashboardService.X is not a function` at runtime.
**Fix:** Either implement the missing dashboard service methods or remove the calls and replace
with direct API client calls.

## Security Concerns

### 4. `ConnectedDashboard.vue` ŌĆö Global `window.applyProfileData()` function
**File:** `src/components/ConnectedDashboard.vue`
**Severity:** Medium
**Issue:** The component exposes `window.applyProfileData = (data) => { ... }` which writes
arbitrary data into reactive state and localStorage. Any script on the page (injected ads,
browser extensions, XSS payloads) can call this function and spoof rank, balance, and points.
**Fix:** Remove the global window function. Use a custom event or Pinia store action instead.

### 5. `ConnectedDashboard.vue` ŌĆö Claim cooldown is client-only
**File:** `src/components/ConnectedDashboard.vue`
**Severity:** Medium
**Issue:** The claim cooldown is enforced entirely via localStorage timestamps. A user can open
DevTools, delete the `valt_next_claim_{email}` key, and immediately claim again without waiting
24 hours. The `claim()` method never calls any API endpoint.
**Fix:** The claim action should call a server endpoint that enforces the cooldown server-side.

### 6. `auth.js` ŌĆö Magic string `'authenticated'` used as placeholder token
**File:** `src/services/auth.js`
**Severity:** Low-Medium
**Issue:** `isAuthenticated()` checks `token !== 'authenticated'` ŌĆö if the string
`'authenticated'` is ever stored as a token value (e.g., from a different code path or manual
localStorage write), auth will silently fail.
**Fix:** Use a clearly namespaced sentinel value or, better, always require a real bearer token
format.

## UX & Reliability Issues

### 7. `ValtDashboard.vue` ŌĆö Send Code has a loading guard but no post-success cooldown
**File:** `src/components/ValtDashboard.vue`
**Severity:** Low
**Issue:** The Send Code button IS disabled while `loading=true` (during the in-flight API call),
preventing double-submission. However, there is no cooldown after a successful send ŌĆö if the
user cancels and re-enters their email, they can send again immediately with no server-enforced
wait. Backend rate limiting is the primary defense here.
**Status:** Partially addressed. Acceptable as long as the backend enforces its own rate limit.

### 8. `ValtDashboard.vue` ŌĆö `signedIn` state is not persisted across page refreshes
**File:** `src/components/ValtDashboard.vue`
**Severity:** Medium
**Issue:** `signedIn` is a local `ref(false)`. On page refresh, it resets to false even if the
user successfully verified their OTP earlier that session. The OTP form reappears on every
load.
**Fix:** On `onMounted`, check `valt_pending_email` and the `claim_available_at` key. If a
valid claim record exists, restore `signedIn = true` automatically.

### 9. `ValtDashboard.vue` ŌĆö `valt_pending_email` cleanup ŌĆö RESOLVED
**File:** `src/components/ValtDashboard.vue`
**Status:** Not a bug. ValtDashboard correctly calls `localStorage.removeItem('valt_pending_email')`
in multiple cleanup paths (on success, on cancel, and on component teardown). Tests confirm
the key is null after successful verification.

### 10. `VerifyTestnet.vue` ŌĆö No email format validation
**File:** `src/components/VerifyTestnet.vue`
**Severity:** Low
**Issue:** The component sends the email value directly to the API without any client-side
format validation. A malformed email (e.g., empty string or `"not-an-email"`) results in an
API round-trip that fails server-side.
**Fix:** Add a simple regex check before submitting: `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`.

### 11. `ConnectedDashboard.vue` ŌĆö Generic localStorage keys risk collision
**File:** `src/components/ConnectedDashboard.vue`
**Severity:** Low
**Issue:** Profile data is stored under bare keys: `rank`, `points`, `profile_balance`,
`connected_apps`, `last_visit`. These are extremely common names that could conflict with other
scripts, browser extensions, or future features stored in the same origin's localStorage.
**Fix:** Namespace all keys with a prefix, e.g., `valt_rank`, `valt_points`, etc.

### 12. `auth.js` ŌĆö `signOut()` swallows API errors silently
**File:** `src/services/auth.js`
**Severity:** Low
**Issue:** `signOut()` wraps the API call in `try/catch` and always clears localStorage even if
the server returns an error. While clearing localStorage is correct, the error is completely
swallowed, making it impossible to detect failed server-side session invalidation.
**Fix:** Log the error or surface it to the caller; at minimum do not silently swallow 500s.

## Architecture & Maintainability

### 13. `ValtDashboard.vue` ŌĆö Component is ~900 lines; should be split
**File:** `src/components/ValtDashboard.vue`
**Severity:** Medium
**Issue:** ValtDashboard contains: navigation bar, OTP email form, code verification form,
countdown logic, toast system, and reward handling ŌĆö all in one 900-line file. This makes it
hard to test individual behaviors in isolation and slows down future refactoring.
**Recommendation:** Extract into smaller composables or sub-components:
- `useOtpFlow.js` composable for send/verify logic
- `useCountdown.js` composable for timer logic
- `ToastNotification.vue` for the toast system
- `OtpEmailForm.vue` for the email + code input UI

### 14. `valt.js` ŌĆö Store holds data that should come from the server
**File:** `src/stores/valt.js`
**Severity:** Low
**Issue:** Several state fields are hardcoded with default values (e.g., `leaderboard` array,
`rewards` array with fixed names/points). These values are never refreshed from the server in
the current implementation.
**Fix:** Remove hardcoded mock data from the store; fetch from API on `initialize()` or
`loadDashboardData()`.

### 15. `router/index.js` ŌĆö All routes have `requiresAuth: false` despite the auth guard existing
**File:** `src/router/index.js`
**Severity:** Low
**Issue:** The navigation guard checks `requiresAuth` per route, but every active route sets it
to `false`. The `/connected-dashboard` route shows signed-in user data but is publicly
accessible without any auth check.
**Fix:** Set `requiresAuth: true` on `/connected-dashboard` and implement the redirect
correctly in the guard once server-side auth is stable.

### 16. `api.js` ŌĆö Mock fallback data lives in the production API client
**File:** `src/services/api.js`
**Severity:** Low
**Issue:** Several VALT endpoints return inline hardcoded mock objects when the real API is
unavailable. Mixing mock data with production API logic makes it hard to know whether any given
response came from the server or was fabricated locally.
**Fix:** Separate mock responses into a dedicated `src/services/api.mock.js` file and only
import it during development/testing.
